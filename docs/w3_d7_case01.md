# w3_d7 Case01：GOAL_SEQUENCE 整数坐标触发参数类型异常，污染回归结果集

## 1. 今日任务背景

今天的任务是完成 **Week3 回归日（w3_d7）** 的三图双向 5-run 小回归，并产出 1 个失败案例分析页。

本次回归使用已经完成的 `w3_d6_run_regression.sh` 作为统一执行脚本，通过 `GOAL_SEQUENCE` 指定交替目标点，按如下思路采集样本：

- A → B
- B → A
- A → B
- B → A
- A → B

这样做的目的是避免“同一点连续发送 goal 导致目标附近瞬时成功”的伪样本，使每一轮结果更接近真实的双向导航运行。

本次回归覆盖三张地图：

- Map1
- Map2
- Map3

其中 Map1 首先建立了稳定的双向回归；随后在 Map2 / Map3 的运行中，出现了部分异常样本。本文件聚焦分析其中一个最有代表性的工程问题：**回归输入未做浮点归一化，导致 ROS2 参数类型异常，从而污染结果集。**

## 2. 本次 case 所在的运行场景

本案例出现在使用 `GOAL_SEQUENCE` 执行 Map2 / Map3 双向回归时。

部分反向目标点采用了整数形式，例如：

- Map2：`-1,2,0.0`
- Map3：`1,3,0.0`

当脚本将这些值传给 `goal_sender` 后，部分样本没有进入正常导航执行阶段，而是在节点初始化阶段直接退出，CSV 中表现为：

- `result=UNKNOWN`
- `goal_sender_rc=1`

这意味着问题不是“导航跑着跑着失败”，而是“样本还没开始导航就已经无效”。

## 3. 现象

在 Map2 / Map3 的初次双向 5-run 回归中，出现了如下异常现象：

1. 某些样本的 `goal_sender` 没有正常进入导航阶段；
2. CSV 中这些样本被记录为 `UNKNOWN`，并伴随 `goal_sender_rc=1`；
3. 异常样本集中出现在反向目标点；
4. 若不查看 run log，仅看 CSV，容易误判为“导航系统本体不稳定”或“方向不对称”。

因此，最初表面看起来像是导航层问题，但从异常触发位置和日志内容看，更像是输入层或脚本层的问题。

## 4. 证据

### 4.1 证据一：终端报错原文

异常样本运行时，终端输出出现如下 traceback：

`InvalidParameterTypeException: Trying to set parameter 'x' to '-1' of type 'INTEGER', expecting type 'DOUBLE'`

以及：

`InvalidParameterTypeException: Trying to set parameter 'x' to '1' of type 'INTEGER', expecting type 'DOUBLE'`

这说明 `goal_sender.py` 中参数 `x` 已按 `DOUBLE` 声明，但命令行传入的整数形式值被 ROS2 解析为 `INTEGER`，最终触发类型不匹配异常。

### 4.2 证据二：异常样本分布具有规律性

异常集中出现在整数形式坐标上：

- Map2 的 `-1,2,0.0`
- Map3 的 `1,3,0.0`

而带小数点的目标点（例如 `-6.2,-0.2,0.0`、`6.15,-1.15,0.0`）能够正常执行。

这说明异常不是随机发生的导航失败，而是具有稳定输入触发条件的问题。

### 4.3 证据三：修复后结果恢复正常

将整数坐标统一改写为浮点形式后：

- `-1 -> -1.0`
- `2 -> 2.0`
- `1 -> 1.0`
- `3 -> 3.0`

重新执行双向 5-run 小回归后：

- Map2：5/5 成功
- Map3：5/5 成功

这说明原先的异常样本不是导航能力问题，而是输入格式错误导致的无效样本。

## 5. 定位分析

该问题根因位于**回归脚本输入层**，而不是 Nav2 规划、控制、TF 或地图层。

问题链路如下：

1. `w3_d6_run_regression.sh` 使用 `GOAL_SEQUENCE` 为每一轮提供目标点；
2. 当目标点写成整数形式时，ROS2 将其解析为 `INTEGER`；
3. `goal_sender.py` 中 `x / y / yaw` 参数以 `DOUBLE` 类型声明；
4. 参数类型不匹配，导致 `goal_sender` 节点在初始化阶段直接抛出异常；
5. 该轮样本未进入正常导航过程，却会在 CSV 中表现为 `UNKNOWN / goal_sender_rc=1`；
6. 如果不检查终端和 run log，很容易误把这种样本当成导航层失败样本。

因此，本问题应定性为：

> **输入层错误污染结果集，而不是导航系统本体退化。**

## 6. 修复措施

### 6.1 立即修复

将 `GOAL_SEQUENCE` 中所有整数形式坐标改写为浮点形式，例如：

- `-1,2,0.0` 改为 `-1.0,2.0,0.0`
- `1,3,0.0` 改为 `1.0,3.0,0.0`

这样可以与 `goal_sender.py` 的参数声明类型保持一致。

### 6.2 建议性长期修复

在 `w3_d6_run_regression.sh` 中加入输入归一化逻辑，对纯整数形式自动补齐为浮点格式，例如：

- `1 -> 1.0`
- `-1 -> -1.0`

这样可以从脚本层减少人为输入失误，避免后续回归再次被同类问题污染。

## 7. 工程意义

这个案例最重要的价值，不是“发现了一次报错”，而是确认了一个更根本的工程事实：

> 回归结果不只会被系统行为污染，也会被实验输入污染。  
> 如果不先排除输入层错误，就无法对导航系统本体做出可信判断。

本案例说明，在导航回归中，以下事情必须同时成立，结果才有分析价值：

1. 目标序列本身合理；
2. 每轮样本真正进入导航执行阶段；
3. 输入参数类型与节点声明保持一致；
4. 异常样本能通过日志与终端输出区分“输入错误”和“系统错误”。

## 8. 结论

本次异常的根因不是 Nav2 导航能力退化，而是 `GOAL_SEQUENCE` 中整数坐标触发了 ROS2 参数类型不匹配，导致部分样本在节点初始化阶段即失效。修复坐标格式后，Map2 与 Map3 的双向 5-run 小回归均恢复为 5/5 成功，说明原先的异常属于输入层问题，而非导航层问题。

本案例的直接结论是：

- 回归脚本输入必须做类型一致性检查；
- 对坏样本必须先判断“输入错了”还是“系统错了”；
- 只有在样本有效性成立的前提下，成功率与耗时统计才具有分析意义。
