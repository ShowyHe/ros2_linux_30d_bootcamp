# Week3 Weekly Report：从手动操作到可验证回归

## 1. 本周目标回顾

Week3 的主线目标不是继续堆叠零散命令，而是将前两周已经具备的基础能力推进到一个更接近工程验证的阶段：  
把 Nav2 从“能手动运行、能看现象”，推进到“能检查、能记录、能批量执行、能输出可分析结果”。

围绕这一目标，本周的工作重心可以概括为六个关键词：

- lifecycle
- healthcheck
- bag record
- bag play
- pipeline logging
- regression

也就是说，本周不是单点突破，而是尝试把前面已经学到的 ROS2 CLI、topic / action 证据链、脚本落盘习惯，组织成一套可以复用的验证流程。

## 2. 本周完成内容

### 2.1 lifecycle：从概念理解转向操作剧本

本周首先将 Nav2 lifecycle 从“知道有这个机制”，推进到“知道如何用命令确认状态、如何理解 inactive / active 等状态差异”。

这一阶段的意义不在于多学了几个命令，而在于建立了一个非常关键的工程意识：

> 判断系统是否可用，不能只看 RViz 画面或 launch 是否启动，而要回到节点状态与生命周期本身。

这为后续 healthcheck 脚本的编写提供了基础判断口径。

---

### 2.2 healthcheck：把系统检查固定成脚本入口

在生命周期与关键节点检查基础上，本周完成了 `w3_d2_nav2_healthcheck.sh` 的最小版本，将以下类型的检查脚本化：

- 关键 node 是否存在
- 关键 topic / publisher 是否存在
- 关键服务或状态是否满足运行前提
- 系统不健康时是否能明确返回非 0

这一阶段的价值在于把“经验式观察”推进成了“脚本化检查”。  
今后再进入回归或演示流程时，不需要完全依赖人工逐条敲命令确认，而是可以先用 healthcheck 建立一个统一入口。

它的工程意义是：

> 在执行任务前，先验证系统是否具备执行条件，而不是带病上路。

---

### 2.3 bag record / bag play：把录制与回放纳入工程流程

本周还推进了 bag 录制与回放的工程化整理，包括：

- 明确录制 topic 白名单
- 固定录制命名与目录组织
- 梳理回放时 `/clock`、QoS、工具配合的基本口径

这一部分的作用不是立刻生成漂亮结果，而是为后续“复现实验现场”“回放异常样本”“做证据留存”建立基础。  
这意味着本周开始，验证不再完全依赖实时现场，而逐步具备了复盘能力。

---

### 2.4 pipeline logging：把过程落盘，而不仅是看终端

在 healthcheck、goal_sender、topic / tf 观察等基础上，本周进一步强化了“全链路输出落盘”的习惯。  
这使得回归过程不再只是“终端跑过一次”，而是逐步形成：

- 过程日志
- 每轮 run 的独立 log
- 结果 csv

这一点非常重要，因为后续无论是排错、写 case、还是做面试展示，都需要能够回答一个问题：

> 这次现象有没有证据，证据在哪，能不能复查？

本周这部分工作，本质上是在为后面的 case 分析和回归总结准备证据基础。

---

### 2.5 regression：从单次发 goal 升级为批量回归

本周最关键的推进，是完成了 `w3_d6_run_regression.sh` 的最小可用版，并在 Week3 回归日继续将其升级为支持：

- 输出不同 csv
- 区分不同日志目录
- 交替目标点序列（`GOAL_SEQUENCE`）

这意味着回归不再是“点一次目标看看成不成功”，而是能够：

- 固定目标序列
- 固定日志落盘
- 固定结果格式
- 按 run_id 追溯单轮执行过程

更重要的是，本周对回归口径做了两次关键纠偏。

#### 第一次纠偏：避免同点连发伪成功

一开始如果连续向同一目标点反复发送 goal，机器人可能已经停留在目标附近，后续样本会出现近零耗时“瞬时成功”。这种样本不能视为真实独立导航样本。

因此，回归方式改为：

- A → B
- B → A
- A → B
- B → A
- A → B

通过交替目标点建立真正有意义的双向样本，而不是重复堆叠伪成功。

#### 第二次纠偏：修复输入层参数类型问题

在 Map2 / Map3 的初次双向回归中，整数形式坐标触发了 ROS2 参数类型异常，导致部分样本在进入导航前即失效。  
修复为浮点形式后，Map2 与 Map3 回归恢复正常。

这说明本周不仅在“跑回归”，也在学习如何校正回归本身的口径。

## 3. 本周最终结果

在修复输入问题并采用交替目标点后，本周完成了三图双向 5-run 小回归，结果如下：

### Map1
- 5/5 成功
- 平均耗时约 28.0 s
- 耗时波动最小

Map1 已可以作为当前最稳定的基线图。

### Map2
- 5/5 成功
- 平均耗时约 70.9 s
- 长尾明显，个别样本接近 timeout 上限

Map2 并非不能通过，但其复杂度和耗时波动显著高于 Map1，更适合在后续作为“复杂场景稳定性观察图”。

### Map3
- 5/5 成功
- 平均耗时约 62.3 s
- 稳定性中等，明显好于最初被输入问题污染时的表象结果

Map3 的结果说明：  
在实验口径正确的前提下，原先看似“方向不对称 / 反向失败 / 伪成功”的现象并不来自导航本体，而是来自输入层异常。

## 4. 本周最重要的工程收获

如果只看表面，本周似乎只是多了几个脚本和几份 csv。  
但真正重要的收获不是“文件变多了”，而是方法发生了变化。

### 4.1 从“现象驱动”转向“证据驱动”

过去容易依赖 RViz 画面、终端即时输出和主观判断。  
本周开始，系统状态、单轮运行过程和最终结果都在逐步落盘，形成了可追溯证据链。

### 4.2 从“单次成功”转向“回归有效性”

本周明确认识到，单次成功并不能代表系统稳定。  
回归是否有效，取决于：

- 样本是否独立
- 目标序列是否合理
- 输入是否规范
- 日志与 csv 是否能对应追溯

### 4.3 从“系统问题”转向“分层定位问题”

本周一个非常关键的进步，是开始区分：

- 输入层错误
- 脚本层错误
- 系统就绪问题
- 导航本体问题

例如，Map2 / Map3 初次异常并非 Nav2 退化，而是 `GOAL_SEQUENCE` 坐标类型错误。  
这说明今后看到坏结果时，不能直接把锅甩给导航系统，而要先判断是哪个层级出了问题。

## 5. 本周仍然存在的不足

虽然本周已经完成三图双向 5-run 小回归，但还有几个不足需要明确写出：

### 5.1 结果汇总仍以人工阅读为主
当前 csv 和日志已经成形，但自动统计与自动汇总还不完善，后续还需要进一步固化结果摘要方式。

### 5.2 回归输入归一化尚未完全脚本化
本周已经定位到整数坐标的参数类型问题，但更稳妥的方式仍然是在脚本层增加自动浮点归一化，避免同类问题再次出现。

### 5.3 失败样本库还不够丰富
本周已经形成 1 个典型 case，但目前案例数量仍然少，后续还需要继续积累更有代表性的失败样本与修复过程。

## 6. 本周结论

Week3 的核心成果，不是简单地“又跑通了几次导航”，而是完成了从手动操作到可验证回归的阶段推进：

- 有健康检查入口
- 有回归执行脚本
- 有独立 run 日志
- 有结构化 csv
- 有跨地图结果
- 有针对异常的定位与纠偏

这说明当前工作已经不再停留在“会用 Nav2”，而开始进入“会验证 Nav2、会纠正实验口径、会用证据说明问题”的阶段。  
对于后续求职准备而言，这一周的内容比单纯多做几个启动脚本更有价值，因为它更接近真实工程场景中的导航验证与排障工作流。
